Work this week focused on ensuring the accuracy of specifications and the stability of the development environment.

- **Cannon VM Specification Updates:** The Cannon fault proof VM specification was updated to include the `getrandom` and `eventfd2` syscalls, and `mprotect` was added to the VM's scope. This resolution unblocks pull requests and restores stability to the continuous integration pipeline ([#870](https://github.com/ethereum-optimism/specs/pull/870)).

## üêõ Issues & Triage
Discussions this week tackled protocol design choices for Interop, testing strategies for Supernodes, and active support for node operators.

- **Closed Issues:**
    - **Documentation & Testing:** A documentation issue regarding the `cannonKonaPrestate` requirement for upgrades was closed, indicating the necessary information has been added ([#18095](https://github.com/ethereum-optimism/specs/issues/18095)). Additionally, the issue to re-enable the previously disabled Kona interop action tests was resolved, although they are not yet a blocking check in CI ([#18613](https://github.com/ethereum-optimism/specs/issues/18613)).

- **New & Active Issues:**
    - **Interop Protocol Simplification:** The most significant ongoing discussion revolves around simplifying the Interop specification by removing the need for an additional injected replacement transaction ([#18574](https://github.com/ethereum-optimism/specs/issues/18574)). While the transaction was initially considered important for the fault dispute game, the consensus is shifting towards having the Supernode generate the necessary proof data on-demand, simplifying the core protocol.
    - **Supernode Test Strategy:** A spike is underway to define how to adapt proof action testing for the Supernode architecture ([#18621](https://github.com/ethereum-optimism/specs/issues/18621)). The conversation emphasizes the need for tests to run against a "real" Supernode to properly validate challenger interactions.
    - **Node Synchronization Support:** An active support thread continues for a user experiencing "missing trie node" errors ([#18581](https://github.com/ethereum-optimism/specs/issues/18581)). The issue is being diagnosed as a potential state corruption problem, with troubleshooting steps being provided.
    - **Challenger Correctness:** A new issue was raised to review the trace output of `op-challenger` in a specific failure case, questioning whether the current test assertions are correct when a super root is not found ([#18643](https://github.com/ethereum-optimism/specs/issues/18643)).
