# ethereum-optimism/optimism Weekly Report (Dec 21 - 27, 2025)

## üöÄ Highlights
This week's development was characterized by a significant architectural push to integrate the `op-supernode` into the challenger system, marking progress towards deprecating the `op-supervisor`. A major milestone was achieved with the completion of the `op-node` Light CL "Follow External Source" feature, enhancing node flexibility. The team also released a substantial upgrade for the `op-deployer` and `op-validator` tools. Concurrently, critical bugs were addressed, including a deployment blocker in `op-contracts` and several integration challenges with the `kona-node`, setting the stage for future work on protocol stability and testing.

## üõ†Ô∏è Key Developments
Work this week focused on architectural evolution, tooling upgrades, and critical bug fixes.

- **Supernode Integration & Challenger Enhancements:**
  Progress was made on replacing the `op-supervisor` with the `op-supernode`. A new super root provider for the `op-supernode` was implemented in `op-challenger` ([#18653](https://github.com/ethereum-optimism/optimism/pull/18653)), and the `superroot_atTimestamp` RPC response format was refined for clearer error handling ([#18652](https://github.com/ethereum-optimism/optimism/pull/18652)). This foundational work supports the ongoing transition to a more streamlined architecture.

- **Tooling Upgrades & Maintenance:**
  The `op-deployer` and `op-validator` tools were upgraded to v6.0.0, incorporating new validation tests to ensure functionality ([#18662](https://github.com/ethereum-optimism/optimism/pull/18662)). In a minor optimization, a redundant AI-generated test was skipped to streamline the CI process ([#18671](https://github.com/ethereum-optimism/optimism/pull/18671)).

- **Bug Fixes and Stability Improvements:**
  Several key bugs were resolved to improve system stability and developer experience:
  - A port collision issue in `op-acceptance-tests` that caused "address already in use" errors was fixed ([#18684](https://github.com/ethereum-optimism/optimism/pull/18684)).
  - A compatibility bug affecting `op-preimage` on Windows was addressed by disabling `setnonblock` ([#18687](https://github.com/ethereum-optimism/optimism/pull/18687)).
  - A code comment in `op-node` was clarified to prevent potential misinterpretation following a security report ([#18512](https://github.com/ethereum-optimism/optimism/pull/18512)).

## üêõ Issues & Triage

- **Closed Issues:**
  The most significant closure was the completion of the `op-node` Light CL "Follow External Source" feature ([#18317](https://github.com/ethereum-optimism/optimism/issues/18317)). This allows an `op-node` to mirror an external safe/finalized feed, a key capability for the Superchain ecosystem.

- **New & Active Issues:**
  - **Critical Deployment Bug:** A new issue ([#18685](https://github.com/ethereum-optimism/optimism/issues/18685)) identified a critical bug in `op-contracts` where deployment fails on fresh EVM environments due to the 24KB contract size limit. This represents a potential blocker for new chain deployments.
  - **`kona-node` Challenges:** A cluster of related issues ([#18672](https://github.com/ethereum-optimism/optimism/issues/18672), [#18673](https://github.com/ethereum-optimism/optimism/issues/18673), [#18674](https://github.com/ethereum-optimism/optimism/issues/18674), [#18675](https://github.com/ethereum-optimism/optimism/issues/18675), [#18676](https://github.com/ethereum-optimism/optimism/issues/18676)) was opened, highlighting difficulties with `kona-node` synchronization, sequencer detection, and testing within `op-devstack`.
  - **Architectural Direction:** An issue was created to formally track the update of `vm-runner` to use the `op-supernode` ([#18669](https://github.com/ethereum-optimism/optimism/issues/18669)), reinforcing the strategic move away from the `op-supervisor`.
  - **Ongoing Node Sync Problems:** Discussions continue in [#18581](https://github.com/ethereum-optimism/optimism/issues/18581) around "missing trie node" errors, indicating persistent synchronization challenges for users running `op-geth`.
